🧭 实验思路

整体思路一句话：模拟一个真实用户的记忆场景，看 fact_store 在这个场景里能不能正确回答问题。

把"是不是好用"这个模糊问题拆成两个具体的东西：

  • fact 集——"它已经记住的事"
  • query 集——"被问到的问题"

然后人工标注每个问题正确答案应该是哪几条 fact（这就是 ground truth），最后跑 fact_store.search()，对比"它返回的"和"正确的"差多少。

六步流程

第一步：造 fact 集（"它记住了什么"）

编一组听起来像真人会存的事实，比如"用户偏好用 Sonnet 4.6"、"主机是 zy242"、"用户在北京时区"……第一轮 20 条手编，后续自动合成 100 / 200 / 500 三档规模。每条用 fact_store.add() 真写进数据库。中文 + 英文各一份，因为怀疑中文会差（事后证实）。

第二步：造 query 集 + 标 ground truth（核心）

每个 query 是真实场景下会问的问题，比如：

  • Q1: "用户用什么模型？" → 应该返回那条 Sonnet 4.6 fact
  • Q2: "项目部署在哪？" → 应该返回那条 mkdocs / GitHub Pages fact
  • Q3: "用户在哪个时区？" → 应该返回那条北京时间 fact

关键：每个 query 手动标了 ground truth——"这个问题正确应该返回哪几条 fact ID"。这是评测的根基，没它就只能"看起来对不对"，那是定性不是定量。

{
  "query": "用户用什么模型？",
  "expected_fact_ids": [3, 7]
}

第三步：跑 search，对比答案

每个 query 喂给 fact_store.search()，它返回 top-k。算两个指标：

  • Precision（精度）：返回的 5 条里，几条是对的
  • Recall（召回）：应该返回的 N 条里，找到了几条

举例：返回 [3,7,12,9,5]、ground truth [3,7] → P=2/5=40%，R=2/2=100%。

第四步：四象限 + 规模扫描

光测一组没用，要看变量影响。最初跑了四象限：

              中文 fact    英文 fact
HRR 开           A           B
HRR 关           C           D

后来加规模轴：20 / 100 / 200 / 500 → 共 12 个 condition（你在下面矩阵里看到的就是这个）。

第五步：contradict 单独测（带负样本）

contradict 是 fact_store 的"找矛盾"功能。专门造了 14 对：

  • 7 对真矛盾："用户用 Opus" vs "用户从不用 Opus"——score 应该高
  • 7 对负样本（看起来像但不矛盾）："项目用 Python" vs "用户喜欢 Python"——score 应该低

实测：真矛盾平均分 0.220，负样本平均分 0.239——比真矛盾还高。这就是为什么它在我场景下基本不工作。没有负样本就发现不了这个问题，只测真矛盾会误以为"它能识别"。

第六步：可视化（就是这一页）

72 个 condition × 6 query × 2 指标 = 800+ 个数字，光看 JSON 谁都晕。HTML dashboard 颜色编码（🟢🟡🔴），一眼能看出哪个塌方。

关键设计决策

  1. ground truth 必须人工标。没这个就只能凭感觉评估，凭感觉等于没评估。每条 query 自己想"真合理的答案是什么"，标完才跑实验。
  2. 必须有负样本。contradict 那部分只测真矛盾时看起来还行，加了负样本立刻发现"对所有看似相关的事都给高分，没区分能力"。
  3. 必须扩到大规模。20 条上测出"HRR 没用"是不公平的。100 / 200 / 500 跑完才发现 HRR 在 100 条英文场景真的赢 22%，但 200+ 塌方。这把"HRR 完全没用"修正成"HRR 适用区间窄"。
  4. 中英对照。没拆中英前以为"HRR 普遍有问题"，拆完才发现中文比英文还差一档——entity 抽取的 regex 是全英文写死的。
  5. P/R 评估有 bug 要修。第一版犯了个错：先 top-k 截断、再算 P/R，导致 P 永远 ≤ 100%。修了之后数字才对。这种 bug 不修结论全错。

为什么这套有说服力

它能被任何人复现。所有数据集、脚本、ground truth 标注都在 ~/garden-experiments/memory-benchmark/，公开 push 到 GitHub。任何人对结论存疑，跑一遍就能验证。这就是"基于数据"的真实含义——不是引用了一个 paper、不是 vibe check 几个 case，是有完整的实验设计 + 公开的可复现实施 + 标注好的 ground truth。

fact_store 真实数据 benchmark

本页是用户真实对话场景的 benchmark——区别于 合成数据 dashboard。
数据来自用户实际记忆栈：fact_store 18 条 + memory 6 条 + USER profile 7 条 = 31 条真实 fact。
Query 来自最近 session 里用户实际问过的"找记忆类"问题，手标 ground truth。

🧭 实验思路（同前）

把"fact_store 真的好用吗"翻译成可测的问题：用户日常问的问题，它能不能正确召回应该返回的 fact。不再用合成的"看起来像真实"的数据——直接用用户真实记忆的全部 31 条 + 真实问过的 12 条问题。

四个 condition

  • HRR + 自然语言：用户原话提问（"我之前 sonnet 4.6 不是配过吗，是 1m 还是 200k？"），开 HRR
  • HRR + 蒸馏关键词：把同一个问题压成关键词（"sonnet 4.6 context"），开 HRR
  • noHRR + 自然语言：清空 hrr_vector，强制走 FTS5 fallback，自然语言 query
  • noHRR + 蒸馏关键词：同上，但用关键词 query

🔥 核心发现 1：自然语言 query 召回率 0% / 0%。用户原话提问 fact_store 完全无法回答——12 条真实问题没一条命中。
🔥 核心发现 2：HRR 开关不影响结果（两栏数字相同）。fact_store 的 search() 接口走 FTS5 关键词，HRR 在 search 路径上根本没参与。
🔥 核心发现 3：必须把 query 蒸馏成关键词才有 38% / 29%。但仍有 6/12 条 query 即使关键词版也 0%——长尾事实（agent-reach 路径、纠错历史、文章风格、时区）全 miss。

💡 解读：为什么会这样

1. fact_store.search() 的实质

读了 hermes 源码 plugins/memory/holographic/retrieval.py 后明确：search() 走的是 SQLite FTS5 全文搜索，不走 HRR。HRR 只在 probe()（按 entity 召回）和 reason()（多 entity 交集）里用。所以 HRR 开关对 search() 没影响——这次实验直接证实了这点。

2. 为什么自然语言全 0

FTS5 把整个 query 字符串当 phrase match。"我之前 sonnet 4.6 不是配过吗，是 1m 还是 200k？" 没 fact 里包含完整 phrase——分词器对中文友好程度有限，"我之前"、"是 1m 还是"这种自然语言粘合词不会被识别为关键 token。

3. 为什么蒸馏关键词只能赢一半

fact_store 必须依赖 fact 内容里有精确字面匹配。所以：

  • 命中：query 关键词 ∈ fact content（"feishu cron"、"NewsBot bot daily-briefing"）
  • miss：query 关键词只描述了语义，不出现在 fact 字面里。例：
  • Q3 "agent-reach 安装路径"——fact 1 里写的是"装在 ~/.local/bin/（pipx 安装）"，没"安装路径"四字
  • Q7 "时区 中文"——fact 201 写"北京时区（UTC+8）"，没"时区"作为关键词以独立 token 形式出现（实际可能能 match，但 FTS5 对中文分词的不确定性导致 miss）
  • Q11 "自信先答错 纠正"——fact 11 写"凭印象答错被纠正"，shape 不对应

4. 真正能用 fact_store 的场景

实测命中的 6/12有共同特征：

  • 查询是专有名词或独特字符串：NewsBot、feishu cron、HRR benchmark、Bedrock patch、cookie、app id
  • 这些字符串必然会出现在对应 fact 的内容里
  • 本质是把 fact_store 当关键词 grep用——和 grep -r "NewsBot" memory.txt 等价

5. 真正不能用的场景

  • 用语义/意图查询而非字面字符串（"我之前...来着"）
  • 查抽象偏好/教训（"用户喜欢什么风格"）
  • 查多个分散 fact组成的主题（"自信先答错"涉及 4 条 fact，但每条用词都不一样）

📌 对用户的实际建议

给 zy242 的具体建议：fact_store 在你日常对话中不工作。你的实际工作记忆是：
  • memory tool（USER.md + MEMORY.md）—— 这两个文件每个 turn 直接拼进 system prompt，agent 自动看到
  • session_search—— 跨 session 找历史，agent 主动搜，LLM 总结后注入
  • fact_store—— 只有当 agent 有意识地用关键词 grep 时才有用

该不该继续往 fact_store 加 fact？

低优先级。除非该 fact 满足：

  • 包含独特专有名词（NewsBot、cli_aa8f549aae385bcd 这种）
  • 是明确事实不是抽象偏好
  • 未来你或 agent 可能会用关键词查询

抽象偏好（"用户讨厌方案对比"）该进 USER.md，不是 fact_store。

# 能用算法解决的不要靠模型推理

!!! abstract "一句话"
    **现在的大模型很聪明，能搞定很多事。但"能搞定"不等于"应该让它搞定"——速度、成本、稳定性三笔账算下来，把工作下沉到算法和数据结构永远更划算。**

---

## 一个反直觉的判断

最近聊 AI agent 记忆系统时，我意识到一个 counterintuitive 的认知：

> **模型再聪明，也不应该让它做能用算法干的事**。

听起来像在贬低大模型，但其实正相反——这是为了**更好地用大模型**。模型该做的是判断、推理、对话；不该做的是"在一堆文件里 grep 关键词"、"取两个集合的交集"、"找出共享主语但内容矛盾的两条记录"。这些事算法毫秒级搞定还更准。

让模型干算法的活，付的不只是 token 钱——是**速度、成本、稳定性的三重代价**。

---

## 三笔账：用一个真实例子算清楚

我在 [Hermes 四层记忆栈](../tech/hermes-memory-stack.md) 那篇里详细对比过两个方案。这里压缩成核心场景：

**问题**：我有几条散落的笔记——飞书主 profile（含 App ID）、Hermes cron 配置（HERMES_CRON_TIMEOUT）、NewsBot（同时是飞书 app + cron 任务）。我问"**飞书相关的 cron 任务有哪些？**"

正确答案只有 NewsBot 那一条——它是飞书 + cron 的精确交集。

### 方案 A：把笔记写成 Markdown，让模型现场推理

存：`feishu.md`（含飞书主 profile + NewsBot）、`cron.md`（含 cron 配置）。

agent 工作流：
1. 看目录，判断该读哪些文件
2. `read_file feishu.md` 全文（200 行）
3. `read_file cron.md` 全文（80 行）
4. 在两个文件里 grep 对方关键词
5. 综合判断哪条是交集

**这条路能 work**——但代价是：

| 指标 | 实际成本 |
|---|---|
| 时间 | 秒级（多次 read_file + 推理） |
| Token | 几千上万（读全文，90% 是噪音） |
| 稳定性 | 看模型当下发挥（漏读哪个文件就漏召回） |

### 方案 B：每条笔记打多维 tag，让算法处理

存：每条笔记附带 `[飞书, cron, NewsBot]` 这种多重 tag，存进结构化存储 + 向量索引。

agent 工作流：
1. 一行命令 `reason(["飞书", "cron"])`
2. 算法逐条算 `min(sim_飞书, sim_cron)`，按分数排序
3. 返回结果

**同样 work，但成本完全不同**：

| 指标 | 实际成本 |
|---|---|
| 时间 | 毫秒级（单次 SQL + numpy 矢量化） |
| Token | 几百（只输出几条 fact 的简短结论） |
| 稳定性 | 算法保证，结果可重现 |

---

## 三笔账深挖

### 第一笔：速度

模型推理是 RTT 级（一次 API 往返几百毫秒到几秒）。算法是 CPU 操作（numpy 矢量化扫 1000 条数据也就毫秒）。

**100 倍以上的差距**。看起来一次查询差几秒不算啥，但 agent 在一次任务里要做几十次"想想该读啥/grep 啥/筛啥"的判断——**累积起来用户感知非常明显**。

我的实际体感：以前查"飞书相关 cron"要等 5-10 秒（模型推理 + 多次 read_file），现在 0.5 秒返回结果。这对**对话流畅度**是质变。

### 第二笔：Token 成本

模型推理的成本结构是 `输入 token × 单价 + 输出 token × 单价 × 4`（输出贵）。

让模型 read_file 全文找答案 = **几千上万输入 token**。这些 token 大部分是噪音（你查"飞书 + cron"，但 feishu.md 里 90% 的内容跟 cron 无关），模型还得自己过滤。

更糟的是：**这些噪音 token 不是无害的占位符**——它们进入 attention，**稀释模型对真正重要信息的注意力**。塞进去越多无关上下文，简单任务都会变笨。这是大语言模型的根本限制：context window 越大不代表用得越好。

算法方案：输入参数几十个 token，输出几条 fact 几百个 token。**两个量级的成本差**。

### 第三笔：稳定性

这是最关键的一笔，也是最容易被忽略的。

**模型推理结果不可重现**——同一个问题问两次可能得到不同答案，因为：

- temperature 带来随机性
- 上下文略有不同就走不同推理路径
- 长文档里的关键句可能被注意力错过
- 弱模型可能根本没意识到要做这个推理

agent 工作流里依赖模型推理的环节越多，**整体可靠性是各环节可靠性的乘积**——任何一环失误整个任务就废。

算法方案：tag 打了就是打了，`reason()` 一定扫到所有 fact，结果完全可重现。**调试时知道为什么这次没召回**（因为没打那个 tag），不需要怀疑"是不是模型今天状态不好"。

## 这个判断的更广应用

不只是记忆系统。这个判断在 AI agent 工程里到处适用：

**找文件**：`grep -r` 比让模型 `read_file` 全文找快 100 倍。
**算交集**：set operations 比让模型读两份内容判断哪些重合稳。
**排序**：`sorted(..., key=...)` 比让模型口头列"你认为哪个更重要"准。
**去重**：`set()` 或哈希比让模型读完判断"哪些重复了"准且省 token。
**找异常**：阈值判断 + 规则比让模型读监控数据找异常稳。
**结构化提取**：JSON schema + 验证比让模型自由文本格式化稳。
**重试**：try/except + 重试逻辑比让模型判断"这次要不要重试"准。

通用模式是：

> **任何一件事，如果能用确定性算法表达，就不要让模型现场推理**。
>
> 模型的价值在**判断和创造**——什么时候该做某件事、做的内容是什么。**做的过程**留给算法。

## 反例：什么时候该让模型推理

为了避免被理解成"反对一切让模型干活"，再讲一下边界：

**该让模型推理的事**：

- **判断分类暧昧**："这条用户反馈是 bug 还是 feature request" — 没有清晰规则
- **生成自然语言**："给这段日志写个用户能看懂的摘要" — 算法做不到
- **理解隐性意图**："用户说'再给我推几个'，是要更多结果还是要换一批" — 上下文判断
- **新颖组合**："设计一个有创意的 onboarding 流程" — 创造性

这些事是**模型独有的能力**——算法一辈子也写不出来。

但记忆检索、文件搜索、集合运算、排序去重——这些是**模型不擅长但算法擅长的**。让模型做这些是**误用工具**：付了 GPU 推理的钱，得到 grep 的结果。

判断公式：

> **能写成 Python 一行的，写 Python；写不成的，让模型上**。

---

## 为什么这个判断现在变得重要

**因为模型确实变聪明了**。

GPT-4 Turbo / Claude Opus 这一代真的能"硬扛"很多 agent 工作——给它一堆文件让它找答案，它能找到。给它一段含错信息的对话历史，它能识别。给它两个集合让它取交集，它能算出来。

**正是因为它能扛**，开发者很容易陷入"模型啥都行，全交给它就好了"的诱惑。结果：

- 一个 agent 任务跑 30 秒
- 单次任务花 50 个 token 单价
- 同一个问题两次答案不一样

**模型变强不是省 engineering 的理由，恰恰是更值得 engineering 的理由**——因为它强到能掩盖架构的问题，让你以为系统设计很好，其实是模型在替你擦屁股。

好的 AI agent 工程，是**让模型只在它独有的能力上发挥**——其它环节用算法、用结构化数据、用确定性规则把它框住。这样：

- 可预测的部分用算法，**可解释、可调试、可重现**
- 不可预测但需要的部分用模型，**得到模型独有的判断和创造**
- 整体系统：**速度快、成本低、行为稳定**

---

## 一句话总结

> **AI agent 工程的优雅，不是让模型显得很聪明，是让算法显得很聪明，模型只在该出场的地方出场**。

这是一个一旦内化就回不去的认知。下次设计 agent 工作流，每一步问一句："**这一步真的需要模型推理吗？还是用算法更稳？**"

通常答案都是后者。
